# Yandex Wiki API: полевые заметки

Находки проверены живьём на продакшн-организации Яндекс 360. Org-нейтральные
probe-скрипты в [`scripts/`](../scripts/) (`probe_api*.sh`, `smoke.sh`) — живая
документация этого поведения, их можно перезапустить на своей организации
(креды через env-переменные или файл `$SECRETS`; вывод скриптов идёт в `raw/`,
который в gitignore, потому что содержит реальные данные организации).

## Эндпоинт поиска (`POST /v1/search`)

Эндпоинт недокументирован, но публичен — это тот же бэкенд, что обслуживает строку
поиска в веб-интерфейсе Вики. Первым его обнаружил и опубликовал
[slartus/mcp-yandex-wiki](https://github.com/slartus/mcp-yandex-wiki), что напрямую
вдохновило `page_search`; здесь эти находки независимо перепроверены и расширены
(например, `page_size` принимает до 50, а не до 10).

- Максимум `page_size` — **50** (0, отрицательные или >50 → HTTP 400). Тулза ограничивает значение до 1–50 на клиенте.
- **Серверной пагинации и фильтрации нет** — `page`/`offset`/`limit` и любые body-параметры
  секций/типов игнорируются, а `total_pages` всегда 1 (или 0, когда пусто).
  Аргументы тулзы `slug_prefix` и `result_type` применяются на клиенте после получения результатов.
- `total_documents` всегда равен числу возвращённых результатов — это **не** глобальный счётчик совпадений.
- Результаты двух типов: **`page`** (относительный url; тулза нормализует его в абсолютную
  ссылку на базе `WIKI_WEB_BASE_URL`) и **`file`** (абсолютная ссылка на скачивание `...?download=1`).
- Запросы `"в кавычках"` работают и дают фразовый поиск; `-минус` и булевы операторы игнорируются.

## Авторизация, скоупы и права

- **OAuth-скоупы Wiki API не проверяет** — токен только с `wiki:read` всё равно может
  писать. Read-only гарантируется только нерегистрацией write-тулзов
  (`WIKI_READ_ONLY=true`). *Credit: первым публично сообщил
  [slartus/mcp-yandex-wiki](https://github.com/slartus/mcp-yandex-wiki), здесь подтверждено независимо.*
- **HTTP 403 — про права пользователя**, а не про скоупы токена: например, readonly
  системные страницы, принадлежащие `yandex360-wiki` (по slartus, см. выше).
- Источника заголовка организации два: `X-Org-Id` (Яндекс 360) и `X-Cloud-Org-Id`
  (Yandex Cloud); сервер выставляет один из них по `WIKI_ORG_ID`/`WIKI_CLOUD_ORG_ID`.

## Страницы

- **Любой `POST /pages/{id}` обновляет `modified_at`**, даже с пустым телом — страница
  помечается изменённой (по slartus, см. выше).
- **API ревизий/истории/обратных ссылок нет** — сценарии «кто ссылается сюда» невозможны.
- `created_at`/`modified_at`/`comments_count`/`is_readonly` — не топ-левел поля страницы;
  их нужно запрашивать через `page_get` с `fields=["attributes"]`.
- `GET /pages/{id}/resources?q=` — единственный серверный *текстовый* фильтр во всём API
  (поиск по названию среди вложений/гридов одной страницы) — доступен через `page_get_resources`.

## Гриды

- Эндпоинты мутаций гридов используют optimistic locking: требуют актуальную `revision`
  и отклоняют запрос, если она устарела.
- Wire-формат `default_sort` — список словарей с одним ключом, например
  `[{"status": "asc"}]`. Тулза `grid_update` принимает более дружелюбный формат
  `[{"column": "status", "direction": "asc"}]` и конвертирует его сама.
- `grid_add_columns` требует `required` у каждой колонки — API это валидирует.
- `grid_copy` асинхронный: API возвращает метаданные операции, а не готовую копию грида.

## Ошибки и лимиты

- Ошибки приходят в **двух форматах envelope** (`message` строка-или-null плюс
  `details`, либо список плюс `level`); клиент парсит оба и пробрасывает собственное
  сообщение API в `WikiApiError`.
- Заголовков rate-limit нет (`X-RateLimit-*`/`Retry-After` отсутствуют).
